# Tech Selection Research

ライブラリ・フレームワーク・SDK・API・SaaS・ツールチェインなど、特定の技術スタックの選定を支援するスキル。複数候補を客観的な軸で比較し、ユーザーの要件・制約に沿った推奨案を提示する。

## このスキルが扱うもの・扱わないもの

- **扱う**: 状態管理ライブラリの比較、ORM選定、認証SaaS比較、UIフレームワーク選定、テストランナー選定、ロギング基盤の選定、CIサービス選定、決済プロバイダ比較など。「同じ目的を達成できる複数の技術選択肢」の比較。
- **扱わない**:
  - サービス・プロダクト全体の競合調査 → `competitive-research` スキル
  - 「マイクロサービス vs モノリス」等のアーキテクチャパターン選定（このスキルの設計はライブラリ/SaaS粒度の選定に最適化されている）
  - ユーザーが既に1つに決めている技術の使い方解説（その場合は context7 を直接使う）

迷ったら短く確認する。例: 「これは認証SaaSの比較リサーチでよいですか？それとも認証アーキテクチャ全体の設計議論を含めますか？」

## リサーチの進め方

### 1. 要件・制約の抽出

リサーチに入る前に、ユーザーの依頼から次の情報を抽出する。これがないと比較軸が定まらず、汎用的なネット記事の要約になってしまう。

- **技術カテゴリ**: 何の選定か（例: ORM / 状態管理 / 認証）
- **動作環境**: 言語・ランタイム・フレームワーク・OS・ホスティング先
- **既存スタック**: 連携が必要なライブラリやインフラ（package.json / pyproject.toml / Gemfile などを読むのが速い）
- **規模・制約**: 想定ユーザー数、データ量、レイテンシ要件、予算、商用利用可否
- **チーム事情**: チームの得意言語、習熟度、運用体制

文脈から読み取れる項目は確認せず進めてよい。読み取れない項目のうち**結論を左右しそうなもの**だけ、ユーザーに一問一答で短く確認する。すべて聞き返すと体験が悪い。

### 2. 候補の洗い出し

候補を**5〜8個**ほど洗い出す。少なすぎると比較にならず、多すぎると焦点がぼやける。デファクトの2〜3個に加えて、最近台頭してきた選択肢や、用途特化のニッチ候補を1〜2個混ぜると視野が広がる。

検索パターン:
- `best <category> for <language/framework> 2026`
- `<library A> vs <library B>` / `alternatives to <library>`
- `<category> comparison <year>` / `<category> benchmark`
- 日本語でも検索する（日本語コミュニティ事情・実例記事が拾える）

### 3. 情報ソースの使い分け

複数のソースを組み合わせる。1つに頼ると偏りが出る。

| ソース | 取りに行く情報 |
|---|---|
| **WebSearch** | 比較記事、ベンチマーク記事、ユーザー体験談、最新の動向、移行事例 |
| **context7** (`mcp__context7__resolve-library-id` → `query-docs`) | 公式ドキュメントの正確な記述、APIの形、サポート機能の一次情報 |
| **GitHub** (WebFetchで `https://github.com/<org>/<repo>` や API) | Star数、最新コミット日、リリース頻度、open/closed issue比率、メンテナ数、ライセンス |
| **competitive-research スキル** | 技術選定の結論がプロダクト戦略やUXに依存するケース（例: 認証SaaSを選ぶ際に他社プロダクトでの採用事例を見たい） |

context7はAPI仕様を取りに行くときの一次情報として優秀なので、機能比較を書くときに WebSearch だけで済ませず必ず併用する。WebSearch の記事は古いことが多く、現行バージョンの仕様と乖離していることがある。

### 4. 評価軸の決定

最低限、以下の**必須軸**は全候補について埋める。これらは候補を見比べる土台になる軸で、欠けると比較が成立しない。

- **コミュニティの規模**: GitHub Star数、npm/PyPI等のダウンロード数、Stack Overflowのタグ数、主要な採用企業
- **学習コスト・ドキュメントの豊富さ**: 公式ドキュメントの質、チュートリアルの多寡、日本語資料の有無、概念の複雑さ

加えて、ユーザーの要件・制約から導かれる**コンテキスト依存の軸**を追加する。例:
- パフォーマンス要件があるならベンチマーク値
- 商用利用が前提ならライセンス
- 長期運用前提ならメンテナンス状況（最終リリース日・コミッター数）
- 既存スタックとの統合性
- 価格（SaaSの場合）

軸はユーザーの要件と無関係に増やさない。比較表が肥大化して読みにくくなる。

### 5. 比較表＋推奨案の作成

[出力フォーマット](#出力フォーマット)に従って整理する。比較表は事実、推奨案は解釈、と役割を分ける。読み手が「比較表だけ見て自分で判断する」道も「推奨案だけ読んで採用する」道も選べるようにする。

## 出力フォーマット

必ずこの構造に沿って書く。セクションごとに役割が決まっていて、欠けると判断材料が偏る。

```markdown
# [技術カテゴリ] 技術選定リサーチ

## 調査概要
- 調査日: YYYY-MM-DD
- 対象カテゴリ: [例: Node.js向け認証ライブラリ]
- 前提条件: [動作環境・既存スタック・規模感など重要な制約]
- 評価軸: [選定した軸を箇条書き。必須軸＋コンテキスト依存軸]

## サマリー
[3〜5行。市場の構図、本命候補、推奨の方向性を一目で伝える]

## 候補一覧

### [候補1]
- **公式**: [URL]
- **GitHub**: [URL] / Star: XXk / 最終リリース: YYYY-MM-DD
- **ライセンス**: [MIT / Apache-2.0 / 商用ライセンスなど]
- **概要**: [1〜2文]
- **強み**: [箇条書き]
- **弱み・注意点**: [箇条書き]
- **代表的な採用事例**: [わかる範囲で]

### [候補2]
...

## 比較表

| 評価軸 | 候補1 | 候補2 | 候補3 |
|---|---|---|---|
| コミュニティの規模 | Star: 30k / npm DL: 5M/週 | ... | ... |
| 学習コスト・ドキュメント | 公式◎、日本語△、概念シンプル | ... | ... |
| [コンテキスト依存軸1] | ... | ... | ... |
| [コンテキスト依存軸2] | ... | ... | ... |

## 推奨案

### 第一推奨: [候補名]
**理由**: [ユーザーの要件・制約に紐づけて、なぜこれかを2〜4文で説明]
**想定される懸念**: [採用した場合のリスクや注意点]

- 情報の鮮度: [YYYY-MM-DD時点の情報。料金・バージョン等は要再確認]
```

## 振る舞いの指針

- **数字は出典つきで**: Star数やDL数は調査時点の値を書く。`Star: 30k (2026-05時点)` のように日付を添える。数字を出せないなら「コミュニティ規模: 大/中/小」のような相対評価でよい（推測値を断定で書かない）。
- **比較表の空欄を埋めるために嘘をつかない**: 確認できなかったセルは `要確認` / `不明` と書く。これは弱さではなく誠実さで、ユーザーが追加リサーチ箇所を判断できる。
- **推奨案は要件に紐づける**: 「人気だから」「新しいから」では選定にならない。ユーザーの制約のどれを満たすからこれを推すのか、を明示する。
- **両論併記しすぎない**: 推奨案を曖昧にすると意思決定の助けにならない。第一推奨は1つに絞る。同点なら同点と書く（その判断は自分で下す）。
- **コードスニペットは要件に直結するときだけ**: API形状の違いが採用判断を左右する場合は最小コード比較を入れる。形式が違うだけで本質的な差がない場合は入れない。
- **競合スキルとの境界**: サービス自体の競合比較（料金・UX・市場ポジショニング）が主目的なら `competitive-research` を使う。技術スタックとしての適合性が主目的ならこのスキルを使う。両方の性格が混ざっているなら、ユーザーに目的を確認する。
